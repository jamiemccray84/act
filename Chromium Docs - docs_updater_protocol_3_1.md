href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#Introduction"><span></span></a><a class="h" name="introduction" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#introduction"><span></span></a>Introduction</h2><p>The Omaha protocol defines the interactions between a software updater (an “Omaha Client”) and a cloud update infrastructure. The cloud infrastructure contains both an update control server (an “Omaha Server”) and a collection of plain HTTP servers or CDNs (collectively, the “Download Servers”). The protocol is an application-layer protocol on top of HTTP.</p><p>Omaha clients interact with the servers in a three-phase <strong>session</strong>. Each step in the session involves at least one successful HTTP transaction.</p><ol><li><strong>The Update Check</strong>: The client transmits what applications it has installed that may be eligible for update to the server. The server replies with whether an update is available for each application. If no updates are available, this is the end of the update session.</li><li><strong>The Download(s)</strong>: The client downloads the updates from the download servers. It then executes downloaded installers or its own code to apply these updates.</li><li><strong>The Ping-Back(s)</strong>: The client transmits data about its activities back to the server, including whether the updates were successful, which download URLs were used, and any error codes encountered.</li></ol><p>The update check is an HTTP POST with a JSON data body. The response has a JSON data body. The integrity of the update check is protected by <a href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/cup.md">CUP</a>, even in the presence of compromised TLS.</p><p>The downloads are HTTP GET requests. The client is free to use any HTTP technology to fetch the downloads, for example using Range requests.</p><p>The ping-back is an HTTP POST with a JSON data body. The response may contain a JSON data body acknowledging the received data, but should be ignored by the client unless CUP is in use on this request.</p><h2><a class="h" name="Concepts" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#Concepts"><span></span></a><a class="h" name="concepts" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#concepts"><span></span></a>Concepts</h2><h3><a class="h" name="Session-ID" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#Session-ID"><span></span></a><a class="h" name="session-id" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#session-id"><span></span></a>Session ID</h3><p>All requests in a session are linked by a unique <strong>session ID</strong>. Session IDs are random strings with at least 128 bits of entropy. There must be no relationship between a previous session ID and the next session ID. In the update check and ping-back requests the session id is transmitted in the POST body (details below). In the download requests it is transmitted as an HTTP header (details below).</p><h3><a class="h" name="Request-ID" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#Request-ID"><span></span></a><a class="h" name="request-id" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#request-id"><span></span></a>Request ID</h3><p>Each update check has a unique <strong>request ID</strong>. Each ping-back has a different unique request ID. Request IDs are random strings with at least 128 bits of entropy. There must be no relationship between a previous request ID and the next request ID. Request ID allows an Omaha server to deduplicate repeated requests, which have been observed as part of client or proxy retry loops.</p><h3><a class="h" name="Default-Values" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#Default-Values"><span></span></a><a class="h" name="default-values" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#default-values"><span></span></a>Default Values</h3><p>The request bodies of the update check and ping-back requests are JSON. Most members have a default value defined by this protocol. Clients and servers may omit serialization of any members the have default values. Therefore, if a client or server does not send the member, the recipient must assume the default value of that member.</p><p>The default value of any list-valued member is the empty list. The default value of any object-valued member is an empty object (which is equivalent to an object of the appropriate type populated with its default members).</p><p>Compatible clients and servers must be able to tolerate unexpected members.</p><h3><a class="h" name="Version-Numbers" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#Version-Numbers"><span></span></a><a class="h" name="version-numbers" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#version-numbers"><span></span></a>Version Numbers</h3><p>Version numbers are strings of the form A.B.C.D, where A, B, C, and D are base 10 integers between 0 and 2^32 - 1 (inclusive). Suffix elements may be omitted if equal to 0; that is, “1.2” is equal to  “1.2.0.0”.</p><p>Version numbers are ordered. Version A is greater than version B if and only if there some element of version A is greater than the corresponding element of version B, and all the preceding elements are equal. Two versions are equal if any only if all their elements are equal.</p><h3><a class="h" name="Application-IDs" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#Application-IDs"><span></span></a><a class="h" name="application-ids" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#application-ids"><span></span></a>Application IDs</h3><p>An application ID is a string that identifies an application, for example “com.google.chrome”. Application IDs are case-insensitive in the ASCII characters. Application IDs should only contain printable, human-readable characters.</p><h3><a class="h" name="User-Counting" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#User-Counting"><span></span></a><a class="h" name="user-counting" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#user-counting"><span></span></a>User Counting</h3><p>The Omaha protocol is anonymous, but allows servers to compute the number of unique active or update-checking devices. This is accomplished through client-regulated counting.</p><p>The idea of client-regulated counting is to inspect all update check requests received over the last N days and discard all but the first request from each client. The number of remaining requests is equal to the number of unique clients.</p><p>Each response from the server contains a numeric date, representing the date (in the server‘s choice of timezone) that the client’s request was received. The client stores this date, and sends it on the next request to the server. When inspecting the next request, the server can determine whether the date is before (current date - N + 1). If so, this is the first request from the client in the N-day window. Otherwise, this request is a “duplicate” and can be discarded.</p><p>In certain environments (for example, frequently re-imaged VMs in internet cafes), it is likely that the client may fail to update the date of the last transaction. To avoid overcounting such clients, a technique called ping-freshness is used. A ping-freshness value is a random string with at least 128 bits of entropy that is written into the client‘s data store alongside the server’s date. This value is sent along with the date in the next request. If the server receives multiple requests with the same ping-freshness, this is a signal that a machine's state has been cloned or reset and that the dates in the request are not trustworthy. A new ping-freshness value is written any time a new date is written.</p><p>In addition to counting the total population for each application, the Omaha protocol counts the active user population for each application. When an application is actively used, it signals this to the client, and the client will transmit the “last reported active” date to the server on the next update check (in addition to the “last checked” date). These dates are separately writeable and therefore need separate ping-freshness values as well.</p><p>The “last reported active” and “last checked” date may vary per managed application, and the client must maintain separate dates (and separate ping-freshness values) per application.</p><h3><a class="h" name="Differential-Updates" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#Differential-Updates"><span></span></a><a class="h" name="differential-updates" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#differential-updates"><span></span></a>Differential Updates</h3><p>The Omaha protocol supports differential updates. A differential update achieves better compression by relying on information (binaries) the client already has.</p><p>A version number is usually insufficient to identify the binaries the client already has, since they may vary by architecture, platform, or other variables while retaining the same official version number. Therefore, the server sends a more precise label with each update payload, which the client reports back in subsequent update checks. This value is called a “differential fingerprint”.</p><p>The server should send a value determined by the hash of the binary (not, for example, a unique ID). In practice, Google's servers always send “1.hash” where “hash” is the SHA256 hash of the update payload.</p><h3><a class="h" name="Extensions-Forward-Compatibility" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#Extensions-Forward-Compatibility"><span></span></a><a class="h" name="extensions-forward-compatibility" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#extensions-forward-compatibility"><span></span></a>Extensions &amp; Forward Compatibility</h3><p>The protocol is extensible via the addition of new object members. Clients must tolerate the existence of members they do not handle. Unofficial or application-specific additional members should be prefixed with an underscore to avoid colliding with later additions to the protocol.</p><p>Additions to the protocol that specify backward-compatible default values (often with a semantic meaning of “unknown”) do not need to increase the protocol version. Removals of values with specified defaults from the protocol do not need to increase the protocol version, since the default value can be assumed by compatible endpoints. All other changes to the protocol may require a new version number.</p><h3><a class="h" name="Timing-Backoff" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#Timing-Backoff"><span></span></a><a class="h" name="timing-backoff" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#timing-backoff"><span></span></a>Timing &amp; Backoff</h3><p>Clients and servers are free to negotiate the rate at which the client conducts update sessions. However, the following best practices are recommended:</p><ul><li>If a client periodically checks according to a time-elapsed scheduled task, (e.g. “check every 5 hours”), those checks should happen at an interval that is coprime with a 24-hour day. This prevents a specific client from being locked to a particular set of times at which it checks for an update.</li><li>As much as possible while respecting a user or administrator's preferences, clients should avoid check policies that rely on specific clock values (e.g. “check at 5:00 AM”). Clocks throughout the world are roughly synchronized and update schedules like this cause abrupt spikes in update load on update infrastructure.</li><li>If a client cannot conduct a check at the scheduled time (it is in an off or standby power state or lacks network connectivity, the client should check as soon as it recovers from that condition (subject to the below). This allows clients who are only briefly connected to reliably update.</li><li>A client doing periodic checks should implement a random chance to postpone a periodic update by a fraction of that period. Client update schedules can become synchronized by server-side or network outages, by image cloning, by synchronized installations, or other effects. This behavior gradually desynchronizes a cohort of clients that have become synchronized for any reason. A 20% chance to delay by 0% to 20% of the period (chosen randomly) is generally good enough.</li><li>Clients should avoid checking at a predictable millisecond within the second, a predictable second within the minute, or a predictable minute within the hour. This both helps resist fingerprinting of the client and smooths load on the server, which might otherwise have spikes of traffic at precisely 1:00:00.000, 1:00:01.000, 1:01:00.000, etc.</li></ul><p>In the case of update check failure, retries are generally permitted. The client is free to attempt an immediate retry or a retry after a delay. At scale, even a single retry may double load on the update server regardless of any client delay (unless the client simply delays to its next normal periodic update check. Clients must not retry if they successfully receive a CUP-validated response from the server, regardless of the contents. Servers may protect themselves in the case of overload by issuing an X-Retry-After HTTP header, detailed <a href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#headers-update-check-response">below</a>.</p><p>For download requets, clients should prefer to fall back to subsequent download URLs rather than attempting retries on a particular URL.</p><h3><a class="h" name="Safe-JSON-Prefixes" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#Safe-JSON-Prefixes"><span></span></a><a class="h" name="safe-json-prefixes" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#safe-json-prefixes"><span></span></a>Safe JSON Prefixes</h3><p>JSON responses from the server are prefixed with <code class="code">)]}'\n</code>, where \n indicates a new line character. This prevents them from being naively executed as a script in the context of the update server's origin.</p><h2><a class="h" name="Update-Checks" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#Update-Checks"><span></span></a><a class="h" name="update-checks" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#update-checks"><span></span></a>Update Checks</h2><p>An update check is the first phase of an Omaha session. When attempting a download of a differential or full package, clients must attempt downloads from each URL of the appropriate type in the specified order, falling back to the next URL if a TCP or HTTP error is encountered. A 4xx or 5xx HTTP response qualifies as an error that justifies a fallback. A successful download of a file that fails to hash to the expected hash or has an unexpected size also qualifies. Other network errors may also qualify.</p><ul><li><code class="code">url</code>: A list of <code class="code">url</code> objects.</li></ul><h4><a class="h" name="Update-Checks-Body-Update-Check-Response-Objects-Update-Check-Response-11" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#Update-Checks-Body-Update-Check-Response-Objects-Update-Check-Response-11"><span></span></a><a class="h" name="update-checks-body-update-check-response-objects-update-check-response-11" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#update-checks-body-update-check-response-objects-update-check-response-11"><span></span></a><code class="code">url</code> Objects (Update Check Response)</h4><p>A url object describes a URL prefix. It has the following members:</p><ul><li><code class="code">codebase</code>: An absolute URL prefix. Presence of this member indicates a full download URL. This member will be present if and only if <code class="code">codebasediff</code> is not present. To create a full URL, the value must be concatenated with a <code class="code">package.name</code> member.</li><li><code class="code">codebasediff</code>: An absolute URL prefix. Presence of this member indicates a differential download URL. This member will be present if and only if <code class="code">codebase</code> is not present. To create a full URL, the value must be concatenated with a <code class="code">package.namediff</code> member.</li></ul><h4><a class="h" name="Update-Checks-Body-Update-Check-Response-Objects-Update-Check-Response-12" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#Update-Checks-Body-Update-Check-Response-Objects-Update-Check-Response-12"><span></span></a><a class="h" name="update-checks-body-update-check-response-objects-update-check-response-12" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#update-checks-body-update-check-response-objects-update-check-response-12"><span></span></a><code class="code">action</code> Objects (Update Check Response)</h4><p>An action represents a task that the update client must conduct after application of the update (if an update is provided), or immediately (if no update is provided). It has the following members:</p><ul><li><code class="code">type</code>: Indicates the type of action. Known values:<ul><li>“run”: This action is an instruction to run one of the application's executables. If the run action is served along with an update, and the update fails, the run action should not be executed. If the run action is served without an update, it should be executed unconditionally.</li><li>“launchcmd”: This action is an instruction to run the application's registered launchcmd command. A launchcmd command is defined as part of the protocol between an Omaha client and an application installer.</li><li>“hideui”: This action is an instruction to hide any UI associated with the updater, and to exit silently without further user input. For example, this can be used in conjunction with a launchcmd action to instruct a metainstaller to launch the installing product and immediately appear to exit.</li></ul></li></ul><p>Additional members may be present, depending on the action type.</p><p>For <code class="code">type == "run"</code>:</p><ul><li><code class="code">run</code>: The path to the executable (relative to the root of the CRX archive that the client was served in this update (if an update was served) or in an update with the associated differential fingerprint (if this response does not contain an update for this application). For other action types, this member may not appear.</li><li><code class="code">arguments</code>: The command line arguments to be passed to the executable, formatted as a single string.</li></ul><hr><h2><a class="h" name="Downloads" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#Downloads"><span></span></a><a class="h" name="downloads" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#downloads"><span></span></a>Downloads</h2><p>Download requests occur when an application update is needed, as a result of a <code class="code">response.app.updatecheck.pipelines.operations.urls</code> member.  Download requests are HTTP GET requests and can use any HTTP implementation.</p><h3><a class="h" name="Request-Headers" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#Request-Headers"><span></span></a><a class="h" name="request-headers" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#request-headers"><span></span></a>Request Headers</h3><p>In addition to the regular HTTP headers, this protocol defines the following headers for the download request:</p><ul><li><code class="code">X-Goog-Update-SessionId</code>: The <a href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#session-ids">session ID</a> of the update session.</li></ul><hr><h2><a class="h" name="Ping_Backs" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#Ping_Backs"><span></span></a><a class="h" name="ping_backs" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#ping_backs"><span></span></a>Ping-Backs</h2><p>Ping-back requests are caused by any install, update, or action attempted during an update session. Ping-backs share a similar structure to update check requests.</p><p>Ping-back requests are fire-and-forget: the client can discard the server‘s response. Ping-back transactions need to be protected by CUP if and only if the client handles the server’s response.</p><p>Ping-back requests can be <strong>bundled</strong> into a single HTTP transaction, or sent immediately as the events triggering the ping-back occurs. Bundling ping-backs reduces QPS to the server, but risks ping-back loss if the client crashes, loses connectivity, or loses power before it transmits the bundle.</p><h3><a class="h" name="Ping_Back-Request-Headers" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#Ping_Back-Request-Headers"><span></span></a><a class="h" name="ping_back-request-headers" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#ping_back-request-headers"><span></span></a>Ping-Back Request Headers</h3><p>Similarly to update-check requests, ping-back requests have additional headers for the purpose of easier DoS rejection.</p><ul><li><code class="code">X-Goog-Update-AppId</code>: A comma-separated list of application IDs included in this request.</li><li><code class="code">X-Goog-Update-Updater</code>: “name-version”, where “name” is the name of the updater as reported in <code class="code">request.updater</code> and “version” is the version of the updater as reported in <code class="code">request.updaterversion</code>.</li></ul><h3><a class="h" name="Ping_Back-Request-Body" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#ping_back-request-body"><span></span></a>Ping-Back Request Body</h3><p>The request body of an ping-back contains a JSON object with the following members:</p><ul><li><code class="code">request</code>: A <code class="code">request</code> object.</li></ul><h4><a class="h" name="Ping_Backs-Ping_Back-Request-Body-Objects-Ping_Back-Request-1" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#Ping_Backs-Ping_Back-Request-Body-Objects-Ping_Back-Request-1"><span></span></a><a class="h" name="ping_backs-ping_back-request-body-objects-ping_back-request-1" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#ping_backs-ping_back-request-body-objects-ping_back-request-1"><span></span></a><code class="code">request</code> Objects (Ping-Back Request)</h4><p>A ping-back <code class="code">request</code> object contains all the same members as a update check <code class="code">request</code> object. Differences occur in the <code class="code">request.app</code> objects.</p><h4><a class="h" name="Ping_Backs-Ping_Back-Request-Body-Objects-Ping_Back-Request-2" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#Ping_Backs-Ping_Back-Request-Body-Objects-Ping_Back-Request-2"><span></span></a><a class="h" name="ping_backs-ping_back-request-body-objects-ping_back-request-2" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#ping_backs-ping_back-request-body-objects-ping_back-request-2"><span></span></a><code class="code">app</code> Objects (Ping-Back Request)</h4><p>A ping-back <code class="code">app</code> object is identical to an update check <code class="code">app</code> object, except for the following differences.</p><p>A ping-back <code class="code">app</code> object cannot contain any of the following members:</p><ul><li><code class="code">data</code></li><li><code class="code">ping</code></li><li><code class="code">updatecheck</code></li></ul><p>A ping-back <code class="code">app</code> additionally contains the following members:</p><ul><li><code class="code">event</code>: a list of <code class="code">event</code> objects.</li></ul><h4><a class="h" name="Ping_Backs-Ping_Back-Request-Body-Objects-Ping_Back-Request-3" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#Ping_Backs-Ping_Back-Request-Body-Objects-Ping_Back-Request-3"><span></span></a><a class="h" name="ping_backs-ping_back-request-body-objects-ping_back-request-3" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#ping_backs-ping_back-request-body-objects-ping_back-request-3"><span></span></a><code class="code">event</code> Objects (Ping-Back Request)</h4><p>An event object represents a specific report about an operation the client attmpted as part of this update session. All events have the following members:</p><ul><li><code class="code">eventtype</code>: The event type is a numeric value indicating the type of the event. It must always be specified by the client. The following values are known:<ul><li>2: An install operation.</li><li>3: An update operation.</li><li>4: An uninstall operation.</li><li>14: A download operation.</li><li>41: An app command completion event.</li><li>42: An action operation.</li></ul></li><li><code class="code">eventresult</code>: The outcome of the operation. Default: 0. Known values:<ul><li>0: error</li><li>1: success</li><li>4: cancelled</li></ul></li><li><code class="code">errorcat</code>: An error category, for use in distinguishing between different classes of error codes. Default: 0. The following values are known:<ul><li>0: No category.</li><li>1: Errors acquiring the download.</li><li>2: Errors during CRX unpacking.</li><li>3: Update client errors during installation.</li><li>4: Errors within the update service itself.</li><li>5: Error during update check.</li><li>7: Application installer errors during installation.</li></ul></li><li><code class="code">errorcode</code>: The error code (if any) of the operation. A client that successfully applies a differential update should send the result both here and in <code class="code">eventresult</code>. A client that attempts and fails a differential update should send the result here, and use <code class="code">eventresult</code> to indicate the outcome of the full update attempt.</li><li><code class="code">differrorcat</code>: As <code class="code">errorcat</code> but for differential updates. Similar to <code class="code">diffresult</code>.</li><li><code class="code">differrorcode</code>: As <code class="code">errorcode</code> but for differential updates. Similar to <code class="code">diffresult</code>.</li><li><code class="code">diffextracode1</code>: As <code class="code">extracode1</code> but for differential updates. Similar to <code class="code">diffresult</code>.</li><li><code class="code">previousfp</code>: The <a href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#differential-updates">differential fingerprint</a> the client had prior to the update, regardless of whether that update was successful.</li><li><code class="code">previousversion</code>: The application version the client had prior to the update, regardless of whether that update was successful.</li></ul><p>For <code class="code">eventtype == 14</code> events:</p><ul><li><code class="code">download_time_ms</code>: The time elapsed between the start of the download and the end of the download, in milliseconds. -1 if unavailable. Default: -1.</li><li><code class="code">downloaded</code>: The number of bytes successfully received from the download server. Default: 0.</li><li><code class="code">downloader</code>: A string identifying the download algorithm / stack. Known values:<ul><li>"" (empty string): Unknown downloader.</li><li>“nsurlsession_background”: MacOS background NSURLSession.</li><li>“bits”: Microsoft BITS.</li><li>“direct”: The Chromium network stack.</li></ul></li><li><code class="code">total</code>: The number of bytes expected to be downloaded. Default: 0.</li><li><code class="code">url</code>: The URL from which the download was attempted.</li></ul><p>For <code class="code">eventtype == 41</code> events:</p><ul><li><code class="code">appcommandid</code>: The id of the app command for which the ping is being sent.</li></ul><p>For <code class="code">eventtype == 42</code> events:</p><ul><li><code class="code">actiontype</code>: The type of the action that caused this event.</li></ul><h3><a class="h" name="Ping_Back-Response-Body" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#Ping_Back-Response-Body"><span></span></a><a class="h" name="ping_back-response-body" href="https://chromium.googlesource.com/chromium/src/+/main/docs/updater/protocol_3_1.md#ping_back-response-body"><span></span></a>Ping-Back Response Body</h3><p>Clients are free to ignore the ping-back response body.
